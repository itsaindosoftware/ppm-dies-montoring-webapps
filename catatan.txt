Kenapa sekarang belum terjadi

Flag #sym:is_4lot_check saat ini baru dipakai untuk tampilan/form, belum jadi hard rule alur proses.
Tombol Start PPM Processing dikontrol dari logic UI di Show.jsx:24 dan Show.jsx:63, tapi tidak ada pengecekan khusus #sym:is_4lot_check.

Di model, lots per cycle memang 4, tapi enforcement wajib maintenance belum ada di state machine DieModel.php.
Langkah yang baik (best practice pabrik otomotif)

Definisikan state baru:
maintenance_required → maintenance_in_progress → maintenance_completed.
Trigger otomatis:
saat lot ke-4 tercapai (untuk die dengan #sym:is_4lot_check), langsung set ke maintenance_required.
Interlock produksi:
selama maintenance_required/in_progress, input produksi untuk die itu diblok atau minimal hard warning + supervisor override.
Ubah aksi di UI:
di tahap ini jangan tampilkan Start PPM Processing, ganti tombol menjadi Start Maintenance.
Wajib evidence:
maintenance tidak boleh complete tanpa checklist, PIC, waktu mulai-selesai, temuan, tindakan, spare part, approval.
Setelah maintenance_completed:
baru lanjutkan alur berikutnya (misalnya transfer atau record PPM sesuai kebijakan plant Anda).
Aturan backend yang wajib (paling penting)

Di endpoint Start PPM Processing:
jika #sym:is_4lot_check = true dan status belum maintenance_completed, return error.
Di endpoint Record PPM:
tambahkan validasi sama, supaya tidak bisa bypass via API/Postman.
Di job/perhitungan status:
ketika mencapai checkpoint 4 lot, set maintenance_required secara otomatis (bukan menunggu user klik manual).
Parameter trigger yang aman

Gunakan basis stroke sejak maintenance/PPM terakhir, bukan akumulasi total lifetime.
Formula praktis:
lots_since_last_maintenance = floor((accumulation_stroke - stroke_at_last_maintenance) / lot_size).
Jika lots_since_last_maintenance >= 4 dan #sym:is_4lot_check true, maka maintenance_required.
